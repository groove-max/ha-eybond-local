[![Open EyeBond Local Card in HACS](https://my.home-assistant.io/badges/hacs_repository.svg)](https://my.home-assistant.io/redirect/hacs_repository/?owner=groove-max&repository=ha-eybond-local-card&category=plugin)

**EyeBond Local** — інтеграція для Home Assistant, яка спілкується напряму з гібридними інверторами через Wi-Fi колектор SmartESS / EyeBond, не звертаючись до хмари виробника.

Ви отримуєте моніторинг у реальному часі, лічильники енергії та керування підтримуваними інверторами — все локальною мережею.

Якщо штатний моніторинг вашого інвертора працює через застосунок SmartESS, це зазвичай найкращий індикатор сумісності. Сам SmartESS при цьому, як правило, може працювати паралельно з EyeBond Local.

> **Зауваження:** Інтеграція активно розвивається. Для деяких інверторів спочатку доведеться надіслати архів підтримки, щоб ми могли підтвердити сумісність і визначити наступний крок. Дивіться [Підтримуване обладнання](#підтримуване-обладнання) та [Як отримати допомогу](#як-отримати-допомогу).

## Що вміє

- **100% локально** — без хмари виробника, без залежності від інтернету.
- **Майстер налаштування** з автопошуком і ручним резервним сценарієм.
- **Безпечно за замовчуванням** — керування лишається тільки для читання, доки система не впевнена в розпізнаванні.
- **Необов'язковий SmartESS cloud assist** — збирає повторно використовувані cloud evidence для діагностики та підказок метаданих, не обходячи локальні запобіжники безпеки.
- **Налаштовуваний інтервал опитування** — від `2` до `3600` секунд, тож дані можна оновлювати помітно частіше, ніж у штатному SmartESS.
- **Готово для панелі енергії Home Assistant** — обчислені лічильники для PV, навантаження, батареї та, на підтримуваних моделях, імпорту/експорту з мережі.
- **Відкрита архітектура** — описи профілів і регістрів у форматі JSON.

## Підтримуване обладнання

EyeBond Local працює з інверторами, для яких штатний моніторинг доступний через **SmartESS**. Це може бути окремий **Wi-Fi колектор SmartESS / EyeBond** або вбудований Wi-Fi модуль, що використовує той самий локальний протокол.

Штатний моніторинг через SmartESS зазвичай продовжує працювати паралельно: EyeBond Local не підміняє його і не заважає йому.

| Протокол / профіль | Стан | Примітки |
|---|---|---|
| **SMG / Modbus** | Підтримується | Локальний протокол/профіль для частини гібридних інверторів; повний моніторинг + перевірене керування |
| **PI30 ASCII** | Підтримується | Протокол родини PI30; повний моніторинг + перевірене керування на підтримуваних варіантах, зокрема PowMr 4.2kW |
| **PI18 ASCII** | Експериментально | Протокол родини PI18; лише на основі офлайн-відтворення, не для реальної експлуатації |

Назви на кшталт **SMG / Modbus** чи **PI30 ASCII** тут означають протокол або внутрішній профіль сумісності, а не точну комерційну назву моделі інвертора.

Для родини **SMG / Modbus** реально перевірений пристрій має марку **Sandisolar** і модель **SD-HYM-4862HWP**. В інтеграції він поки що показується узагальнено як **SMG 6200**, бо поточна локальна Modbus-телеметрія не віддає надійніший сирий ідентифікатор комерційної моделі.

Наприклад, протестований зараз інвертор **PowMr 4.2kW** повертає сирий рядок моделі `VMII-NXPW5KW`, але в інтеграції показується як **PowMr 4.2kW** у межах родини **PI30 ASCII**.

Не бачите свого інвертора? Він усе одно може запрацювати — створіть звернення на GitHub з [архівом підтримки](#як-отримати-допомогу), і ми зможемо оцінити сумісність та, якщо протокол збігається, розширити підтримку.

## Встановлення

### Через HACS (рекомендовано)

1. Відкрийте **HACS → Інтеграції**.
2. Натисніть меню (три крапки) → **Користувацькі репозиторії**.
3. Додайте `https://github.com/groove-max/ha-eybond-local` як користувацький репозиторій у категорії інтеграцій.
4. Знайдіть **EyeBond Local** у списку та натисніть **Завантажити**.
5. Перезапустіть Home Assistant.
6. Перейдіть у **Налаштування → Пристрої та служби → Додати інтеграцію** і знайдіть **EyeBond Local**.

### Ручне встановлення

1. Завантажте останній реліз.
2. Скопіюйте `custom_components/eybond_local/` у каталог `config/custom_components/` свого Home Assistant.
3. Перезапустіть Home Assistant.
4. Додайте інтеграцію через **Налаштування → Пристрої та служби**.

## Покрокове налаштування

Майстер виконує більшість роботи замість вас.

**1. Ласкаво просимо** — оберіть тип підключення і підтвердіть. Майстер одразу показує, з якого інтерфейсу Home Assistant буде сканувати мережу.

<p align="center"><img src="docs/images/setup-01-welcome.png" alt="Екран привітання" width="320"></p>

**2. Сканування мережі** — інтеграція надсилає запит виявлення у локальну мережу. Зазвичай це триває 5–15 секунд.

<p align="center"><img src="docs/images/setup-02-scanning.png" alt="Сканування мережі" width="480"></p>

**3. Виявлені пристрої** — побачите список знайдених колекторів та інверторів зі статусами:

- **Готово** — впевнено розпізнано, безпечно додавати.
- **Перегляд** — знайдено, але впевненість низька.
- **Лише колектор** — колектор відповів, але інвертор ще не ідентифіковано.

<p align="center"><img src="docs/images/setup-03-detected-devices.png" alt="Виявлені пристрої" width="320"></p>

**4. Підтвердження** — перевірте модель, серійний номер і драйвер у зведеній таблиці, потім підтвердьте.

<p align="center"><img src="docs/images/setup-04-confirm.png" alt="Підтвердження виявлення" width="320"></p>

Якщо автопошук нічого не знаходить, оберіть **Ручне налаштування** на екрані результатів. Для цього зазвичай потрібно знати локальну IP-адресу Wi-Fi модуля або колектора; найчастіше її можна подивитися у вебінтерфейсі роутера, до якого підключено інвертор. Розширені параметри, як-от порти, виявлення та підтримання з'єднання, сховані в окремий згортуваний блок, тож ви бачите лише найнеобхідніше.

<p align="center"><img src="docs/images/setup-manual.png" alt="Ручне налаштування" width="360"></p>

> **Порада:** Тримайте Home Assistant і колектор в **одній підмережі**, якщо хочете користуватися автопошуком, бо широкомовне виявлення зазвичай не проходить через роутери.

### Ручне налаштування для Remote / NAT

Більшості користувачів ці розширені поля не потрібні. Вони призначені для сценаріїв, де колектор знаходиться віддалено і має підключатися назад через VPN-маршрут або проброс портів.

Використовуйте цю можливість лише тоді, коли звичайного локального виявлення в одній підмережі недостатньо. Якщо Home Assistant і колектор знаходяться в одній локальній мережі, advertised поля краще залишити порожніми.

- **Інтерфейс Home Assistant / локальна IP-адреса слухача**: локальна IPv4-адреса, до якої Home Assistant прив'язує TCP-слухач.
- **IP-адреса колектора**: пряма IPv4-адреса для unicast-перевірки. Для remote/NAT сценаріїв вкажіть публічну або проброшену IPv4-адресу, яка веде до колектора.
- **Локальний TCP-порт**: TCP-слухач на стороні Home Assistant. Типове значення: `8899`.
- **Advertised callback IP**: IPv4-адреса, яка записується в `set>server=...`. Залиште поле порожнім, щоб оголошувати локальну IP-адресу слухача. Заповнюйте його лише тоді, коли колектор має повертатися через іншу VPN/публічну/NAT-доступну адресу.
- **Advertised callback TCP port**: TCP-порт, який записується в `set>server=...`. Залиште поле порожнім, щоб оголошувати локальний TCP-порт. Змінюйте лише якщо зовнішній проброшений порт відрізняється від локального.
- **UDP-порт виявлення**: порт redirect-пакетів discovery для колектора. Типове значення: `58899`.
- **Цільова адреса виявлення**: broadcast підмережі для локальної мережі або конкретна remote/NAT IPv4-адреса для unicast discovery.
- **Інтервал виявлення / інтервал heartbeat**: як часто Home Assistant повторно оголошує себе і надсилає keep-alive після підключення колектора.

Для повного покрокового прикладу з VPN і пробросом портів дивіться [Remote / NAT Setup Guide](docs/REMOTE_SETUP.md).

## Що ви отримаєте

Після налаштування інтеграція додає один пристрій з такими сутностями:

- **Сенсори** — вимірювання інвертора, PV, батареї та мережі: потужність, напруга, струм, температура, частота.
- **Налаштування частоти оновлення** — інтервал опитування можна змінити в межах від `2` до `3600` секунд у параметрах інтеграції.
- **Лічильники енергії** — обчислені значення в `kWh` для PV, навантаження та заряду/розряду батареї, а також імпорту/експорту з мережі на моделях, де доступний показник потужності мережі. Їх можна одразу додати до **панелі енергії Home Assistant**.
- **Бінарні сенсори** — режим роботи, помилки, аварії, стан заряджання.
- **Керування** — `number`, `select`, `switch`, `button` для підтримуваних налаштувань: ліміти заряду, вихідний режим, звуковий сигнал тощо. За замовчуванням приховані, доки не підтверджено, що це безпечно.
- **Діагностика** — стан з'єднання, відповідність драйвера, рівень підтримки, окремий експорт SmartESS cloud evidence і кнопка експорту **архіву підтримки**.

## SmartESS cloud assist

Коли локальне виявлення доходить лише до стану "є колектор" або до низької впевненості, EyeBond Local може додатково звернутися до SmartESS cloud для цього ж ідентифікатора колектора і зберегти повторно використовувані JSON cloud evidence у `/config/eybond_local/cloud_evidence/`.

- **Облікові дані використовуються лише для живого запиту** — інтеграція не зберігає постійну cloud-сесію SmartESS.
- **Cloud evidence допомагають діагностиці та плануванню метаданих** — самі по собі вони не відкривають локальне керування записом.
- **Створити архів підтримки** може автоматично включити вже збережені дані або оновити їх прямо під час побудови ZIP.
- **Експортувати SmartESS cloud evidence** — це окрема розширена дія, якщо потрібні самі дані для перегляду або експериментальних метаданих.
- **Файли evidence зберігаються на диску, доки ви не видалите їх вручну**. Для запису автоматично використовується найновіший відповідний файл.

## Як отримати допомогу

Якщо щось не працює, найшвидший шлях:

1. Відкрийте сторінку інтеграції в Home Assistant, натисніть кнопку налаштування і перейдіть до розділу **Діагностика та експериментальні метадані**.
2. Натисніть **Створити архів підтримки**.
3. Створіть [звернення на GitHub](https://github.com/groove-max/ha-eybond-local/issues) і прикріпіть згенерований ZIP.

Архів підтримки містить анонімізований знімок стану інвертора, читання регістрів і результати розпізнавання. Якщо відповідні SmartESS cloud evidence вже збережені, архів підхопить їх автоматично, а на тому самому екрані можна оновити ці дані прямо перед записом ZIP. Зазвичай цього достатньо, щоб зрозуміти сумісність і визначити наступний крок. Якщо виявиться інший протокол або нестандартна реалізація, може знадобитися додаткова перевірка чи більше однієї ітерації.

### Шаблони звернень

- **Звіт про помилку (Bug Report)** — для відтворюваних регресій на вже підтримуваному обладнанні.
- **Діагностика обладнання (Support Archive / Hardware Diagnostics)** — для непідтримуваного обладнання, проблем з налаштуванням і часткової підтримки. Завжди прикріплюйте згенерований ZIP.
- **Запит на покращення (Feature Request)** — для нової підтримки обладнання чи поліпшення зручності користування.

## Усунення проблем

| Проблема | Що спробувати |
|---|---|
| Автопошук нічого не знаходить | Скористайтеся пунктом **Змінити інтерфейс сканування** і виберіть інший мережевий інтерфейс Home Assistant. Якщо переходите до **Ручного налаштування**, заздалегідь знайдіть локальну IP-адресу Wi-Fi модуля або колектора в інтерфейсі роутера. |
| Зависає на статусі **Лише колектор** | Колектор відповів, але інтеграція ще не змогла впевнено визначити протокол, профіль або точну модель інвертора. Надішліть архів підтримки. |
| Сенсори лишаються недоступними | Переконайтесь, що колектор у тій самій підмережі, що й Home Assistant, і що TCP `8899` / UDP `58899` не заблоковані. |
| Віддалений колектор відповідає, але не підключається назад | Насамперед перевірте **Advertised callback IP** і **Advertised callback TCP port**. Вони мають збігатися з реально досяжною для колектора адресою та проброшеним TCP-портом. |
| Remote/NAT підключення працює нестабільно | Якщо можливо, використовуйте VPN замість "чистого" NAT, особливо коли одна зі сторін за CGNAT або маршрутизатори ненадійно перенаправляють UDP/TCP. |
| Немає елементів керування | У режимі **Авто** керування з'являється лише за високої впевненості розпізнавання і для перевірених можливостей. Якщо, наприклад, інвертор працює за протоколом PI30 і моніторинг уже є, але точну модель не розпізнано, можна в параметрах інтеграції відкрити **Параметри роботи** і вибрати режим **Повний контроль**. Це ручний обхід захисту, який відкриває всі команди без додаткової перевірки, тому використовуйте його лише на власний ризик і бажано після створення архіву підтримки. |

## Структура репозиторію

Коротка орієнтація для тих, хто переглядає вихідний код. Повні нотатки для розробників — у [CONTRIBUTING.md](CONTRIBUTING.md).

- `custom_components/eybond_local/` — вихідний код інтеграції
- `custom_components/eybond_local/profiles/` — декларативні профілі можливостей у форматі JSON
- `custom_components/eybond_local/register_schemas/` — описи регістрів для читання у форматі JSON
- `docs/` — публічна документація та згенеровані звіти
- `tools/` — утиліти командного рядка для діагностики, локальної роботи з фікстурами й валідації
- `.local/fixtures/catalog/` — локальні replay-фікстури, що не потрапляють у git
- `tests/` — модульні й регресійні тести

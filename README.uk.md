> **Картка-компаньйон для дашборду:** Поєднайте інтеграцію з [EyeBond Local Card](https://github.com/groove-max/ha-eybond-local-card), щоб одразу отримати готовий Lovelace-інтерфейс з анімованою схемою потоків енергії та графіками історії.

**EyeBond Local** — інтеграція для Home Assistant, яка спілкується напряму з гібридними інверторами через Wi-Fi колектор SmartESS / EyeBond, не звертаючись до хмари виробника.

Ви отримуєте моніторинг у реальному часі, лічильники енергії та керування підтримуваними інверторами — все локальною мережею.

Якщо штатний моніторинг вашого інвертора працює через застосунок SmartESS, це зазвичай найкращий індикатор сумісності. Сам SmartESS при цьому, як правило, може працювати паралельно з EyeBond Local.

> **Зауваження:** Інтеграція активно розвивається. Для деяких інверторів спочатку доведеться надіслати архів підтримки, щоб ми могли підтвердити сумісність і визначити наступний крок. Дивіться [Підтримуване обладнання](#підтримуване-обладнання) та [Як отримати допомогу](#як-отримати-допомогу).

## Що вміє

- **100% локально** — без хмари виробника, без залежності від інтернету.
- **Майстер налаштування** з автопошуком і ручним резервним сценарієм.
- **Безпечно за замовчуванням** — керування лишається тільки для читання, доки система не впевнена в розпізнаванні.
- **Необов'язкова звірка через SmartESS** — збирає повторно використовувані хмарні дані для діагностики та підказок щодо метаданих, не обходячи локальні запобіжники безпеки.
- **Явне підтвердження запису** — зміни не-дійових параметрів тепер мають підтвердитися негайним readback, а не завершуються мовчазним успіхом, якщо значення не закріпилося.
- **Налаштовуваний інтервал опитування** — від `2` до `3600` секунд, тож дані можна оновлювати помітно частіше, ніж у штатному SmartESS.
- **Готово для панелі енергії Home Assistant** — обчислені лічильники для PV, навантаження, батареї та, на підтримуваних моделях, імпорту/експорту з мережі.
- **Відкрита архітектура** — описи профілів і регістрів у форматі JSON.

## Підтримуване обладнання

EyeBond Local працює з інверторами, для яких штатний моніторинг доступний через **SmartESS**. Це може бути окремий **Wi-Fi колектор SmartESS / EyeBond** або вбудований Wi-Fi модуль, що використовує той самий локальний протокол.

Штатний моніторинг через SmartESS зазвичай продовжує працювати паралельно: EyeBond Local не підміняє його і не заважає йому.

| Комерційна модель / клас обладнання | Внутрішній профіль роботи | Стан | Що це означає |
|---|---|---|---|
| **Sandisolar SD-HYM-4862HWP** | перевірений типовий профіль `modbus_smg` | Підтримується | Повний моніторинг і перевірене керування. У поточному інтерфейсі цей пристрій поки що показується узагальнено як `SMG 6200`, бо локальний протокол передає номінальну потужність, але не дає надійної комерційної назви моделі. |
| **Anenji ANJ-11KW-48V-WIFI-P** | окремий профіль `modbus_smg` для цієї моделі | Підтримується | Працює розширений вбудований моніторинг, зокрема PV1/PV2, дата й час інвертора, а також власні денні та загальні лічильники генерації. Керування вже перевірене на реальному обладнанні, тому перевірені елементи можуть з'являтися і в звичайному автоматичному режимі, коли впевненість розпізнавання висока. |
| **Anenji 4200 (Protocol 1)** | окремий профіль `anenji_4200_protocol_1` | Часткова підтримка | Моніторинг працює за документованим протоколом 1 для SMG-подібних пристроїв, включно з `power_flow_status` і службовими даними про пристрій та налаштування. Розпізнавання поки що не вважається достатньо надійним для автоматичного ввімкнення керування, а сам запис ще не перевірено на реальному обладнанні, тому елементи керування показуються лише в режимі повного доступу. |
| **PowMr 4.2kW** (сирий ідентифікатор `VMII-NXPW5KW`) | драйвер `pi30` з профілем сумісності SmartESS `0925` | Підтримується | Повний моніторинг і перевірене керування для цього обладнання. |
| **Невідомий, але явно SMG-подібний інвертор** | `modbus_smg` `family_fallback` | Лише читання | Використовується, коли інвертор явно належить до родини SMG, але точну модель ще не підтверджено. Моніторинг лишається доступним, але вбудоване керування навмисно вимкнене до появи перевіреної моделі. |
| **Обладнання родини PI18** | експериментальна реалізація `pi18` | Експериментально | Поки що перевірено лише на збережених даних. Це корисно для досліджень і фікстур, але ще не той рівень підтримки, який варто вважати готовим для звичайного користувача. |

У документації використовуються три різні типи назв, і це не одне й те саме:

- **Комерційна модель**: назва на корпусі інвертора, наприклад `Sandisolar SD-HYM-4862HWP`, `Anenji ANJ-11KW-48V-WIFI-P` або `PowMr 4.2kW`.
- **Внутрішній профіль роботи**: спосіб, яким інтеграція спілкується з пристроєм, наприклад `modbus_smg`, `pi30` або `pi18`.
- **Профіль сумісності**: набір описів, які пояснюють інтеграції, що саме можна читати й чим можна керувати. Наприклад, SmartESS `0925`.

Внутрішній спосіб роботи може лишатися узагальненим навіть тоді, коли комерційну модель уже знаємо. Наприклад, перевірений Sandisolar-пристрій і далі працює через типовий профіль `modbus_smg`, бо саме його підтверджують локальні дані з пристрою, хоча сам протокол не віддає надійної комерційної назви моделі.

Не бачите свого інвертора? Він усе одно може запрацювати — створіть звернення на GitHub з [архівом підтримки](#як-отримати-допомогу), і ми зможемо оцінити сумісність та, якщо протокол збігається, розширити підтримку.

## Встановлення

### Через HACS (рекомендовано)

1. Відкрийте **HACS → Інтеграції**.
2. Натисніть меню (три крапки) → **Користувацькі репозиторії**.
3. Додайте `https://github.com/groove-max/ha-eybond-local` як користувацький репозиторій у категорії інтеграцій.
4. Знайдіть **EyeBond Local** у списку та натисніть **Завантажити**.
5. Перезапустіть Home Assistant.
6. Перейдіть у **Налаштування → Пристрої та служби → Додати інтеграцію** і знайдіть **EyeBond Local**.

### Ручне встановлення

1. Завантажте останній реліз.
2. Скопіюйте `custom_components/eybond_local/` у каталог `config/custom_components/` свого Home Assistant.
3. Перезапустіть Home Assistant.
4. Додайте інтеграцію через **Налаштування → Пристрої та служби**.

## Покрокове налаштування

Майстер виконує більшість роботи замість вас.

**1. Ласкаво просимо** — оберіть тип підключення і підтвердіть. Майстер одразу показує, з якого інтерфейсу Home Assistant буде сканувати мережу.

<p align="center"><img src="docs/images/setup-01-welcome.png" alt="Екран привітання" width="320"></p>

**2. Сканування мережі** — стандартний швидкий пошук надсилає широкомовний запит у локальну мережу. Зазвичай це триває 5–15 секунд. Якщо цього недостатньо, на екрані результатів також доступний пункт **Запустити глибоке сканування**: він зберігає той самий початковий крок пошуку, а потім напряму перевіряє решту вибраної IPv4-мережі.

<p align="center"><img src="docs/images/setup-02-scanning.png" alt="Сканування мережі" width="480"></p>

**3. Виявлені пристрої** — побачите список знайдених колекторів та інверторів зі статусами:

- **Готово** — впевнено розпізнано, безпечно додавати.
- **Перегляд** — знайдено, але впевненість низька.
- **Лише колектор** — колектор відповів, але інвертор ще не вдалося точно визначити. Такий результат усе одно можна зберегти як тимчасовий пристрій лише для читання і продовжити діагностику пізніше.

<p align="center"><img src="docs/images/setup-03-detected-devices.png" alt="Виявлені пристрої" width="320"></p>

**4. Підтвердження** — перевірте модель, серійний номер і драйвер у зведеній таблиці, потім підтвердьте.

<p align="center"><img src="docs/images/setup-04-confirm.png" alt="Підтвердження виявлення" width="320"></p>

Якщо швидкий автопошук нічого не знаходить, спочатку спробуйте **Запустити глибоке сканування** на екрані результатів, а вже потім переходьте до **Ручного налаштування**. Глибоке сканування корисне для більших мереж або там, де широкомовний пошук спрацьовує нестабільно, тому воно може тривати помітно довше за звичайний швидкий пошук. Якщо ж ви переходите до ручного налаштування, зазвичай потрібно знати локальну IP-адресу Wi-Fi модуля або колектора; найчастіше її можна подивитися у вебінтерфейсі роутера, до якого підключено інвертор. Додаткові мережеві параметри сховані в окремому згортуваному блоці, тож на екрані лишається тільки найнеобхідніше.

<p align="center"><img src="docs/images/setup-manual.png" alt="Ручне налаштування" width="360"></p>

> **Порада:** Тримайте Home Assistant і колектор в **одній підмережі**, якщо хочете користуватися автопошуком, бо широкомовне виявлення зазвичай не проходить через роутери.

### Тимчасовий пристрій / EyeBond Setup Pending

Якщо майстер уже може зберегти робоче підключення до колектора, але ще не може повністю підтвердити інвертор, він усе одно може створити **тимчасовий запис лише для читання**. У Home Assistant такий запис з'являється як **EyeBond Setup Pending**.

Це проміжний збережений стан, а не автоматично помилка.

Що це означає:

- Інтеграція вже зберегла вибрані вами параметри підключення та самого колектора.
- Сам колектор може бути вже знайдений, але зворотне TCP-підключення, підтвердження моделі інвертора або обидва ці кроки ще не завершилися.
- Діагностика й дії підтримки доступні відразу, навіть якщо звичайні сенсори ще недоступні.
- Запис може завершити налаштування пізніше самостійно, коли колектор перепідключиться або коли ви повторите пошук чи ручну перевірку.

Що робити далі:

1. Зачекайте трохи, а потім оновіть сторінку пристрою. Якщо колектор щойно переналаштовували, його перезавантаження може допомогти.
2. У звичайній локальній мережі тримайте Home Assistant і колектор в одній підмережі, а потім повторіть швидкий пошук, глибоке сканування або ручну перевірку.
3. Для ручного або віддаленого підключення ще раз перевірте IP-адресу колектора, оголошену адресу зворотного підключення, оголошений порт, а також що TCP `8899` і UDP `58899` не заблоковані.
4. Якщо пристрій і далі лишається у стані очікування, відкрийте **Налаштувати → Діагностика та експериментальні метадані** і створіть **архів підтримки**, перш ніж редагувати локальні чернетки.

### Ручне налаштування для віддаленого підключення / NAT

Більшості користувачів ці розширені поля не потрібні. Вони призначені для сценаріїв, де колектор знаходиться віддалено і має підключатися назад через VPN-маршрут або проброс портів.

Використовуйте цю можливість лише тоді, коли звичайного локального пошуку в одній підмережі недостатньо. Якщо Home Assistant і колектор знаходяться в одній локальній мережі, поля з оголошеною адресою зворотного підключення краще залишити порожніми.

- **Інтерфейс Home Assistant / локальна IP-адреса слухача**: локальна IPv4-адреса, до якої Home Assistant прив'язує TCP-слухач.
- **IP-адреса колектора**: пряма IPv4-адреса для перевірки напряму. Для віддаленого підключення вкажіть публічну або переадресовану IPv4-адресу, яка веде до колектора.
- **Локальний TCP-порт**: TCP-слухач на стороні Home Assistant. Типове значення: `8899`.
- **Advertised callback IP**: IPv4-адреса, яка записується в `set>server=...`. Залиште поле порожнім, якщо колектор має підключатися назад до тієї самої локальної адреси Home Assistant. Заповнюйте його лише тоді, коли колектор має повертатися через іншу VPN-, публічну або доступну через NAT адресу.
- **Advertised callback TCP port**: TCP-порт, який записується в `set>server=...`. Залиште поле порожнім, якщо зовнішній порт збігається з локальним. Змінюйте лише тоді, коли зовнішній переадресований порт відрізняється від локального.
- **UDP-порт виявлення**: порт redirect-пакетів discovery для колектора. Типове значення: `58899`.
- **Цільова адреса виявлення**: широкомовна адреса підмережі для локальної мережі або конкретна IPv4-адреса для прямого звернення до віддаленого колектора.
- **Інтервал виявлення / інтервал heartbeat**: як часто Home Assistant повторно оголошує себе і надсилає сигнали підтримання з'єднання після підключення колектора.

Для повного покрокового прикладу з VPN і пробросом портів дивіться [посібник із віддаленого підключення та NAT](docs/REMOTE_SETUP.md).

## Що ви отримаєте

Після налаштування інтеграція додає один пристрій з такими сутностями:

- **Сенсори** — вимірювання інвертора, PV, батареї та мережі: потужність, напруга, струм, температура, частота.
- **Налаштування частоти оновлення** — інтервал опитування можна змінити в межах від `2` до `3600` секунд у параметрах інтеграції.
- **Лічильники енергії** — обчислені значення в `kWh` для PV, навантаження та заряду/розряду батареї, а також імпорту/експорту з мережі на моделях, де доступний показник потужності мережі. Їх можна одразу додати до **панелі енергії Home Assistant**.
- **Бінарні сенсори** — режим роботи, помилки, аварії, стан заряджання.
- **Керування** — `number`, `select`, `switch`, `button` для підтримуваних налаштувань: ліміти заряду, вихідний режим, звуковий сигнал тощо. На деяких моделях це може також включати окремі службові дії, наприклад **Sync Inverter Clock**. Показ керування все одно залежить від упевненості розпізнавання та стану перевірки, але суто runtime-умови SMG тепер показуються як попередження замість прихованого або локально жорстко заблокованого керування.
- **Діагностика** — стан з'єднання, відповідність драйвера, рівень підтримки, лічильники перепідключень колектора та перезапусків discovery, окремий експорт хмарних даних SmartESS, явні позначки режиму лише для читання за потреби й кнопка експорту **архіву підтримки**.

<p align="center"><img src="docs/images/sensors.png" alt="Сенсори після налаштування" width="320"></p>

На підтримуваному обладнанні сторінка параметрів показує нативні елементи керування Home Assistant для змінюваних налаштувань інвертора.

## Звірка через SmartESS

Коли локальне виявлення доходить лише до стану "є колектор" або до низької впевненості, EyeBond Local може додатково звернутися до хмари SmartESS для цього ж ідентифікатора колектора і зберегти повторно використовувані JSON-дані у `/config/eybond_local/cloud_evidence/`.

- **Облікові дані використовуються лише для живого запиту** — інтеграція не зберігає постійний сеанс SmartESS у хмарі.
- **Хмарні дані допомагають діагностиці та плануванню метаданих** — самі по собі вони не відкривають локальне керування записом.
- **Створити архів підтримки** може автоматично включити вже збережені дані або оновити їх прямо під час побудови ZIP.
- **Експортувати хмарні дані SmartESS** — це окрема розширена дія, якщо потрібні самі дані для перегляду або експериментальних метаданих.
- **Файли з хмарними даними зберігаються на диску, доки ви не видалите їх вручну**. Для запису автоматично використовується найновіший відповідний файл.

## Як отримати допомогу

Якщо щось не працює, найшвидший шлях:

1. Відкрийте сторінку інтеграції в Home Assistant, натисніть кнопку налаштування і перейдіть до розділу **Діагностика та експериментальні метадані**.
2. Натисніть **Створити архів підтримки**.
3. Створіть [звернення на GitHub](https://github.com/groove-max/ha-eybond-local/issues) і прикріпіть згенерований ZIP.

Архів підтримки містить анонімізований знімок стану інвертора, читання регістрів і результати розпізнавання. Якщо відповідні хмарні дані SmartESS уже збережені, архів підхопить їх автоматично, а на тому самому екрані можна оновити ці дані прямо перед записом ZIP. Зазвичай цього достатньо, щоб зрозуміти сумісність і визначити наступний крок. Якщо виявиться інший протокол або нестандартна реалізація, може знадобитися додаткова перевірка чи більше однієї спроби.

### Шаблони звернень

- **Звіт про помилку** — для відтворюваних регресій на вже підтримуваному обладнанні.
- **Діагностика обладнання** — для непідтримуваного обладнання, проблем з налаштуванням і часткової підтримки. Завжди прикріплюйте згенерований ZIP.
- **Запит на покращення** — для нової підтримки обладнання чи поліпшення зручності користування.

## Усунення проблем

| Проблема | Що спробувати |
|---|---|
| Автопошук нічого не знаходить | Спочатку скористайтеся пунктом **Змінити інтерфейс сканування** і виберіть інший мережевий інтерфейс Home Assistant. Якщо швидкий пошук усе одно нічого не дав, запустіть **Глибоке сканування** з екрана результатів. Якщо потім переходите до **Ручного налаштування**, заздалегідь знайдіть локальну IP-адресу Wi-Fi модуля або колектора в інтерфейсі роутера. |
| Пристрій зависає в **EyeBond Setup Pending** | Це тимчасовий збережений стан, а не жорстка помилка. Зачекайте трохи, повторіть пошук або ручну перевірку, а якщо підтвердження інвертора чи зворотне підключення колектора так і не завершуються, створіть архів підтримки. |
| Зависає на статусі **Лише колектор** | Колектор відповів, але інтеграція ще не змогла впевнено визначити протокол, профіль або точну модель інвертора. Надішліть архів підтримки. |
| Сенсори лишаються недоступними | Переконайтесь, що колектор у тій самій підмережі, що й Home Assistant, і що TCP `8899` / UDP `58899` не заблоковані. |
| Запис наче прийнявся, але значення одразу повернулося назад | EyeBond Local тепер вважає це явною помилкою readback, а не мовчазним успіхом. Перевірте в діагностиці лічильники перепідключень і restart'ів collector/discovery, тимчасово зупиніть паралельні записи зі SmartESS або штатного застосунку і повторіть спробу після стабілізації колектора. |
| Віддалений колектор відповідає, але не підключається назад | Насамперед перевірте **Advertised callback IP** і **Advertised callback TCP port**. Вони мають збігатися з реально досяжною для колектора адресою та переадресованим TCP-портом. |
| Віддалене підключення через NAT працює нестабільно | Якщо можливо, використовуйте VPN замість "чистого" NAT, особливо коли одна зі сторін за CGNAT або маршрутизатори ненадійно перенаправляють UDP чи TCP. |
| Немає елементів керування | У режимі **Авто** керування з'являється лише за високої впевненості розпізнавання і для перевірених можливостей. Деякі внутрішні профілі роботи навмисно лишаються тільки для читання, наприклад `modbus_smg/family_fallback`. Якщо, наприклад, інвертор працює за протоколом PI30 і моніторинг уже є, але точну модель не розпізнано, можна в параметрах інтеграції відкрити **Параметри роботи** і вибрати режим **Повний контроль**. Це ручний обхід захисту, який відкриває всі команди без додаткової перевірки, тому використовуйте його лише на власний ризик і бажано після створення архіву підтримки. |

## Структура репозиторію

Коротка орієнтація для тих, хто переглядає вихідний код. Повні нотатки для розробників — у [CONTRIBUTING.md](CONTRIBUTING.md).

- `custom_components/eybond_local/` — вихідний код інтеграції
- `custom_components/eybond_local/profiles/` — декларативні профілі можливостей у форматі JSON
- `custom_components/eybond_local/register_schemas/` — описи регістрів для читання у форматі JSON
- `docs/` — публічна документація та згенеровані звіти
- `tools/` — утиліти командного рядка для діагностики, локальної роботи з фікстурами й валідації
- `.local/fixtures/catalog/` — локальні фікстури для відтворення, що не потрапляють у git
- `tests/` — модульні й регресійні тести
